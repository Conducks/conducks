# Frontend — i18n Protocol

> Multi-language is a structural decision. If it's enabled, it must be complete.

---

**I18N-1 — Explicit initialization** `[severity: high]`
Do not implement i18n routing or message infrastructure unless the user explicitly requests it. If requested, use a standardized library (e.g., `next-intl`) and define the supported locales in `conducks.config.json`.

**I18N-2 — Mandatory Translation Keys** `[severity: high]`
Every user-visible string must be moved to a message dictionary (`src/i18n/messages/*.json`). Hardcoded strings in `page.tsx` or components are not permitted.

```tsx
/* correct */
<h1>{t('auth.login.title')}</h1>

/* wrong */
<h1>Welcome back to the portal</h1>
```

**I18N-3 — Locale Routing Model** `[severity: medium]`
- **Multi-language**: Use localized route segments (`/[locale]/...`).
- **Single-language**: Use feature-based folders directly in `src/app/`.
- Ensure query parameters and state are preserved during locale switching.

**I18N-4 — No Hollow Keys** `[severity: high]`
Message dictionaries must be organized by domain namespaces (e.g., `auth.errors.*`, `billing.invoice.*`). Generic keys like `title`, `submit`, or `error` that are shared across unrelated domains are banned to prevent translation collisions.

---

## Technical Checklist

- [ ] Supported locales defined in config
- [ ] Fallback locale set for unsupported requests
- [ ] Locale switcher preserves current path and params
- [ ] RTL support included (if applicable)
- [ ] No hardcoded strings in components
